# Phase J — Environment Compatibility & Hardening

**Intended release version:** `v2.3.0` — the whole phase ships together.

**Theme.** Pyve plays nicely with the realities of the user's shell and OS. Three sub-themes:

1. **asdf / direnv coexistence.** Disable asdf's Python-plugin reshim-after-`pip install` hook inside pyve-managed projects so venv-installed CLIs resolve via `.venv/bin` instead of being silently dispatched through `~/.asdf/shims/`. See [pyve-asdf-reshim-bug-brief.md](pyve-asdf-reshim-bug-brief.md) for the full root-cause analysis.
2. **Rip Category A deprecation paths.** Phase H shipped two delegation-with-warning paths (`pyve testenv --init|--install|--purge`, `pyve python-version <ver>`). The userbase is small and the delegation code (alias mapping + warning emission + re-dispatch + tests) is the expensive part. Remove them. The Category B hard-error legacy-flag catches (three-line entries in `legacy_flag_error()`) stay — they cost nothing and give precise hints to users and LLMs coming from stale docs.
3. **Preemptive bash 3.2 compatibility audit.** macOS ships `/bin/bash` at 3.2.57. Phase H had two bash-4+ slips caught only after merge (H.e.7a `declare -A`, H.e.9h `mapfile`). Add grep-invariant tests to catch future slips pre-commit.

All three sub-themes are "pyve interoperates cleanly with the realities around it." Grouping them gives Phase J a coherent narrative at a sensible size (~6 stories).

## Gap Analysis

### asdf reshim coexistence

**What exists today.**

- `lib/env_detect.sh::detect_version_manager` already detects asdf and returns `"asdf"` in `$VERSION_MANAGER`.
- `.envrc` is generated by two branches in `pyve.sh` (venv at ~line 1042, micromamba at ~line 1076). Both branches write a fixed template; neither inspects `$VERSION_MANAGER`.
- `pyve run` dispatches to `.venv/bin/<cmd>` (venv) or `micromamba run -p <prefix>` (micromamba). Neither injects any asdf-aware env var.

**What's missing.**

- No mechanism to disable asdf's Python-plugin reshim after `pip install` inside a pyve venv. The bug reproduces any time a user installs a new CLI into a pyve-managed venv on a system with asdf + zsh.
- No defense for `--no-direnv` users (where `.envrc` doesn't exist).

**What this phase delivers.**

- `pyve init` appends `export ASDF_PYTHON_PLUGIN_DISABLE_RESHIM=1` to the generated `.envrc` when asdf is detected. Idempotent across reinit. No user prompt — silent info line.
- `pyve run` injects the same env var into its subprocess environment when asdf is detected. Covers `--no-direnv` / explicit-execution workflows.
- `pyve purge` removes `.envrc` as it already does; no additional cleanup needed.
- No changes to the user's shell rc files. No sentinel-block machinery in `~/.zshrc`. Everything is project-scoped.

### bash 3.2 compatibility

**What exists today.**

- Pyve supports bash 4+ as its documented minimum (per `tech-spec.md`), but macOS CI runs `/bin/bash` 3.2.57 and surfaces bash-4+ slips.
- Individual stories (H.e.7a, H.e.9h) added ad-hoc grep invariants in their own test files (`test_ui.bats`) after each bug was found.
- No central, enforced grep-invariant test catches the full set of bash-4+ constructs.
- No "smoke-source every `lib/*.sh` under `/bin/bash`" check.

**What's missing.**

- A single `tests/unit/test_bash32_compat.bats` file with one `@test` block per construct — fails loudly if any construct lands in tracked source.
- (Optional) `make check-bash32` target to source every `lib/*.sh` under `/bin/bash` as a smoke test.

**What this phase delivers.**

- A grep-invariant unit test covering the full construct list (`declare -A` / `typeset -A` / `local -A`, `mapfile` / `readarray`, `${var^^}` / `${var,,}` / `${var^}` / `${var,}`, `${var@…}`, `declare -n`, named `coproc`, `**` globstar).
- A full-repo scan (one-time, within the phase) to confirm the current tree is clean. Any surfaced issues become their own small fix tasks under the relevant story.
- Optional `make check-bash32` Makefile target.

## Feature Requirements (new)

### FR-J1: `.envrc` asdf compatibility guard

On `pyve init` (both backends), when `$VERSION_MANAGER == "asdf"` is true, the generated `.envrc` includes:

```sh
# Prevent asdf Python plugin from reshimming venv-installed CLIs.
# See: https://github.com/pointmatic/pyve/blob/main/docs/specs/pyve-asdf-reshim-bug-brief.md
export ASDF_PYTHON_PLUGIN_DISABLE_RESHIM=1
```

- Emitted only when asdf is detected on the system. On pyenv-only or version-manager-absent systems, the block is not written (no pollution).
- Idempotent on reinit: if `.envrc` already exists and contains the sentinel comment (`# Prevent asdf Python plugin from reshimming`), it is not duplicated.
- `pyve init` emits one info line explaining what was added (see FR-J3 for wording).
- Skipped under `--no-direnv` (no `.envrc` is created at all in that path).
- Silent opt-out via `PYVE_NO_ASDF_COMPAT=1` env var; silent opt-in via `PYVE_ASDF_COMPAT=1` (for symmetry and CI ergonomics). CLI flag (`--no-asdf-compat`) is **out of scope** for J — env var is sufficient, and adding a flag commits us to supporting it forever. Revisit if users request it.

### FR-J2: `pyve run` asdf compatibility guard

When `pyve run` executes a subprocess and `$VERSION_MANAGER == "asdf"` is true, it exports `ASDF_PYTHON_PLUGIN_DISABLE_RESHIM=1` into the subprocess environment.

- Covers `--no-direnv` workflows where `.envrc` is absent.
- Covers one-shot CI invocations that don't involve direnv.
- Same `PYVE_NO_ASDF_COMPAT=1` opt-out applies.
- No user-visible output — this is defense-in-depth, not a feature announcement.

### FR-J3: User-facing notice on `pyve init`

When the asdf guard is added to `.envrc` (FR-J1), emit one info line after the "Created .envrc" success line:

```
▸ asdf detected — added ASDF_PYTHON_PLUGIN_DISABLE_RESHIM=1 to .envrc so
  pip-installed CLIs resolve to .venv/bin instead of ~/.asdf/shims.
  (Global pip-installs outside a venv will need `asdf reshim python` afterward.)
```

- Lives below the `.envrc` creation success line, above any later init stages.
- Uses the `lib/ui.sh::info` helper (dimmed cyan arrow).
- One line conceptually, wrapped at ~80 cols.
- Suppressed under `PYVE_NO_ASDF_COMPAT=1` (no block, no line).

### FR-J4: bash 3.2 compatibility invariant test

A new test file `tests/unit/test_bash32_compat.bats` containing one `@test` block per construct in the disallowed set. Each `@test` greps the source tree (`pyve.sh`, `lib/*.sh`, `lib/completion/*`) and fails if any match is found outside the test file itself.

- Construct set: `declare -A`, `typeset -A`, `local -A`, `mapfile`, `readarray`, `${var^^}`, `${var,,}`, `${var^}`, `${var,}`, `${var@…}` (anchored `@[UuLlQqEePpAaKk]`), `declare -n`, `coproc <name>`, `shopt -s globstar`.
- Self-exclusion: the grep patterns must not match themselves in the test file (use anchored patterns or `--exclude` flags).
- Each block's failure message names the construct and the bash 3.2 alternative (e.g., "use flat colon-delimited string instead of `declare -A`").

### FR-J5 (optional): `make check-bash32` smoke target

A Makefile target that invokes `/bin/bash -c 'source lib/foo.sh'` for every `lib/*.sh` file and reports any that fail to source. Complements FR-J4's grep invariants with an "actually works" check.

- Optional — if `/bin/bash` is not at 3.2 on the developer's machine, the target is a no-op with a clear warning. macOS CI provides the real coverage.
- Not wired into default `make test` to keep local cycles fast.

## Technical Changes

### `lib/env_detect.sh`

Add a thin helper:

```sh
is_asdf_active() {
    [[ "${VERSION_MANAGER:-}" == "asdf" ]]
}
```

- Callers: `.envrc` generator, `pyve run` dispatcher, tests.
- Gate by `PYVE_NO_ASDF_COMPAT`: return non-zero when the user has opted out, so all callers short-circuit consistently.

### `pyve.sh`

- **`.envrc` generator (both branches, ~L1042 + ~L1076).** After the existing template body, append the asdf compat block when `is_asdf_active` returns 0. Use heredoc to preserve formatting.
- **`pyve run` dispatcher.** Before executing the subprocess, if `is_asdf_active`, prefix the exec with `env ASDF_PYTHON_PLUGIN_DISABLE_RESHIM=1` (or `export` + exec; prefer `env` prefix for scoping to the single invocation).
- **Reinit idempotency.** The reinit path reads existing `.envrc` content. The grep-for-sentinel check prevents double-appending on `pyve update` / `pyve init --force`.

### `docs/specs/features.md`

- New FR-J1 / FR-J2 / FR-J3 entries under a new section `### FR-J1: asdf Reshim Compatibility` (or similar). Cross-link to `pyve-asdf-reshim-bug-brief.md`.
- Add `PYVE_NO_ASDF_COMPAT` and `PYVE_ASDF_COMPAT` to the Environment Variables table.

### `docs/specs/tech-spec.md`

- New subsection under **Cross-Cutting Concerns**: "asdf/direnv Coexistence (Phase J / v2.3.0)" describing the guard, where it's emitted, and the sentinel-grep-for-idempotency pattern.
- Update **Testing Strategy** to reference `tests/unit/test_bash32_compat.bats`.

### `tests/unit/test_bash32_compat.bats` (new)

One file, ~10–12 `@test` blocks, one per construct. Each runs a targeted `grep -rE <pattern> pyve.sh lib/ lib/completion/` and asserts zero matches.

### `Makefile` (optional, FR-J5)

Add `check-bash32` target. Not wired into default `test` target.

### `CHANGELOG.md`

Single v2.3.0 entry covering the asdf guard + bash-3.2 invariant test + audit results.

## Out of Scope (deferred)

- **Writing to the user's shell rc file.** Rejected in favor of `.envrc`. If a real use case surfaces (users who don't use direnv and want global protection), revisit in a later phase with the project-guide completion sentinel pattern as prior art.
- **CLI flag `--no-asdf-compat`.** Env var is sufficient for v2.3.0. Adding a flag commits to forever-support; revisit if users ask.
- **Fish / nushell / pwsh support.** Pyve is bash/zsh-scoped; direnv handles fish through its own mechanism.
- **Auto-fixing bash-3.2 violations surfaced by FR-J4.** The audit is expected to find the tree clean (H.e.7a and H.e.9h already cleaned up the known issues). If it surfaces anything, the fix becomes its own task within the relevant J story; it does not balloon the phase.
- **Other asdf plugin quirks** (non-Python plugins, shim hash issues outside `pip install`). Scope is specifically the Python-plugin reshim race documented in the bug brief.
- **`pyve check` diagnostic for "asdf detected but `.envrc` guard missing".** Possible future enhancement; not needed for v2.3.0 since pyve itself writes the guard.
